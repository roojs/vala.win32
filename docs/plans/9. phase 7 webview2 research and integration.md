## Purpose

Embed **Microsoft Edge WebView2** (Chromium) in native Win32 windows from Vala, using the same **MinGW cross-compile + optional Wine smoke** workflow as the rest of vala.win32 — while treating **real Windows** as the authoritative runtime target.

This phase answers:

1. Can we **build** a WebView2 host exe with our current toolchain (Linux host, `x86_64-w64-mingw32-gcc`, Meson)?
2. Can we **run** that exe under Wine for dev iteration, or do we need a Windows machine/VM for meaningful testing?
3. What binding shape fits vala.win32 — generated win32json, hand vapi, C plumbing, or a thin wrapper over an existing loader?

---

## Executive summary (research, June 2025)

| Question | Answer | Confidence |
|----------|--------|------------|
| Cross-compile WebView2 host from Linux with MinGW? | **Yes** — headers + COM calls compile; ship `WebView2Loader.dll` (GNU cannot link `WebView2LoaderStatic.lib`) or use a **builtin loader** (see [webview/webview](https://github.com/webview/webview)) | High — multiple projects (webview, webview2-rs, fcitx-contrib/webview) |
| Generate WebView2 vapi from win32json? | **No** — WebView2 is **not** in [win32metadata](https://github.com/microsoft/win32metadata); separate [WebView2 SDK](https://developer.microsoft.com/en-us/microsoft-edge/webview2/) | Definite |
| Run WebView2 under Wine for dev smoke? | **Sometimes** — fragile, version-sensitive, prefix tuning; **not CI-grade** | Medium — community reports + MS backlog |
| Must we switch to building on Windows? | **No for compile/link**; **yes for reliable runtime validation** | High |

**Recommendation:** Keep **Linux + MinGW + Meson** as the primary build path. Add a **small C COM plumbing layer** + hand vapi (or vendored minimal headers). Validate on **Windows 10/11 with Evergreen WebView2 Runtime**. Treat Wine as **best-effort dev smoke** with documented prefix steps — do not block the phase on Wine parity.

## What WebView2 actually is

WebView2 is **not** a Win32 common control like `WC_EDIT`. It is a **COM-based embedding API** on top of a separately installed **Edge Chromium runtime**.

```
  Application .vala
        │
        ▼  valac → C
  Application .c + webview2-plumbing.c   ← COM vtables, event handlers
        │
        ├──► WebView2Loader.dll          ← locates runtime (or builtin loader)
        │
        └──► Evergreen WebView2 Runtime  ← msedgewebview2.exe / EdgeWebView/
             (machine-wide or fixed version folder)
```

### Components (Microsoft terminology)

| Piece | Role | Ship with app? |
|-------|------|----------------|
| **WebView2Loader** | Finds and loads the runtime | Yes — `WebView2Loader.dll` per arch, or static `.lib` (MSVC only) |
| **WebView2.h / COM interfaces** | `ICoreWebView2Environment`, `ICoreWebView2Controller`, … | Headers at compile time only |
| **WebView2 Runtime** | Chromium engine + Edge binaries | Separate redistributable; often preinstalled on Win10/11 |

Official distribution docs: [Files to ship with the app](https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/distribution#files-to-ship-with-the-app).

### Why win32json does not help

Our pipeline (`marlersoft/win32json` → `generate-binding` → `vapi/`) covers **Windows SDK metadata**. WebView2 lives in its **own SDK/NuGet** (`Microsoft.Web.WebView2`). Nothing in `metadata/win32json-api.files` or `gui.filter` can produce `ICoreWebView2*` bindings today.

Phase 7 therefore adds a **parallel track**: vendored SDK headers + hand-maintained vapi (or generated from `.idl`/`.h` in a one-off tool — out of scope unless maintenance cost bites).

## Build feasibility with current vala.win32 mechanism

### What we already have

| Mechanism | WebView2 fit |
|-----------|--------------|
| Linux host, `valac`, Meson | ✅ Same |
| `x86_64-w64-mingw32-gcc` Track A/B | ✅ Same compiler; WebView2 needs **C++14** only if we embed C++ helper code (webview uses C++ internally; our Vala can stay C with C plumbing) |
| `scripts/win32-gui-prologue.h` | ✅ Extend with `#include <WebView2.h>` when SDK path set |
| `--profile=posix` Track A | ✅ Likely best for WebView2 demo (no GLib required for host window) |
| `--profile=gobject` Track B | Optional — only if we want `Win32.WebView` widget with signals |
| win32json regen | ❌ Not applicable |

### MinGW + WebView2Loader (documented constraint)

From [blckngm/webview2](https://github.com/blckngm/webview2) and [webview/webview](https://github.com/webview/webview#ms-webview2-loader):

- **MSVC:** `WebView2LoaderStatic.lib` links cleanly (VS2019+ toolset).
- **MinGW (GNU):** static loader **does not link** — use **`WebView2Loader.dll`** beside the exe, **or** a **builtin loader** implementation (`WEBVIEW_MSWEBVIEW2_BUILTIN_IMPL=1` in webview).

Cross-compile proof: [webview issue #953](https://github.com/webview/webview/issues/953) — `x86_64-w64-mingw32-g++` on Linux produced a working exe when transferred to Windows with runtime installed; failures were often **wrong loader DLL bitness** (x86 vs x64).

**Action for vala.win32:** Meson copies `WebView2Loader.dll` (x64) next to demo exe, same pattern as Track B copying GLib DLLs from `mingw-libs/`.

### SDK acquisition (build-time)

Options (pick one in spike):

1. **NuGet extract** — `Microsoft.Web.WebView2` → `build/native/include/WebView2.h`, `runtimes/win-x64/native/WebView2Loader.dll`
2. **Fixed SDK zip** in `build/vendor/webview2/` (gitignored) with pin file `metadata/webview2-sdk-ref.txt`
3. **Submodule** of a minimal header pack (no runtime — runtime stays OS install)

No change to `vendor-win32json.sh`.

### Vala binding shape (proposed)

**Tier 1 — Spike (Phase 7a):**

- `src/webview2-plumbing.c` — `CreateCoreWebView2EnvironmentWithOptions`, create controller, resize on `WM_SIZE`, navigate to `https://example.com`
- Minimal `vapi/webview2.vapi` — environment builder + controller + `Navigate` only
- `examples/webview2-demo.vala` — Track A style: `Win32.Window` or raw `CreateWindowExW` + child hosting rect

**Tier 2 — Ergonomic widget (Phase 7b, optional):**

- `Win32.WebView` in widget catalog — **not** from win32json; hand profile in `metadata/widget-conventions.json` pointing at plumbing
- Signals: `navigation-starting`, `navigation-completed`, `web-message-received` (subset)

**Tier 3 — Out of scope here:**

- Full `ICoreWebView2*` surface, WebView2 WinForms/WPF, fixed-version runtime bundling

### COM / callbacks in Vala

WebView2 is event-heavy (`ICoreWebView2NavigationStartingEventHandler`, …). Vala can implement COM interfaces but it is verbose and easy to get wrong. **Prefer C plumbing for vtable stubs** (same philosophy as future `WndProc` relay) and expose a narrow Vala API.

## Wine feasibility (runtime, not compile)

### Official stance

Microsoft tracks Wine/Proton as a **low-priority feature request** ([#3127](https://github.com/MicrosoftEdge/WebView2Feedback/issues/3127)). No supported Wine path from Microsoft.

### Community experience (2024–2025)

Reported working setups share these traits:

| Factor | Typical working pattern | Failure mode |
|--------|-------------------------|--------------|
| Wine version | **≥ 9.2** for installer to finish | Installer hangs / `int $3` in setup |
| Prefix | **win32** or win64; **Windows 7** reported in winecfg for some apps | Black/white webview area on Win10/11 preset |
| Runtime version | Older builds (**~92.x**, **109.x**) more often cited as working | **>109** often black screen even when “installed” |
| Edge install | Manual Edge/WebView2 in prefix; disable Edge Update | Update services loop / crash |
| GPU | `--disable-gpu`, SwiftShader ANGLE flags in some Lutris scripts | Flashing, blank content |

References: [WebView2Feedback #3127](https://github.com/MicrosoftEdge/WebView2Feedback/issues/3127), [winetricks #2226](https://github.com/Winetricks/winetricks/issues/2226), [Wine AppDB test](https://appdb.winehq.org/objectManager.php?sClass=version&iId=42414), [pilot51 Lutris gist](https://gist.github.com/pilot51/e6c52d6b99562db5d074e8519122f20a) (runtime downgrade to 92.x).

### Implication for vala.win32

| Use case | Wine OK? |
|----------|----------|
| Compile + link demo exe | ✅ Already works (MinGW) |
| Smoke: exe starts, window appears | ✅ Likely |
| Smoke: web content renders | ⚠️ **Best-effort** — may need prefix recipe, old runtime, GPU flags |
| CI gate for WebView2 | ❌ **Do not** — flaky |
| Primary QA | ✅ **Real Windows** + Evergreen runtime |

We can document `scripts/run-wine-webview2.sh` (prefix setup, optional runtime pin) mirroring `scripts/run-wine.sh` — but Phase 8 CI should **not** require WebView2 under Wine.

### Relation to “Wine runs Edge”

Installing **Edge browser** in a prefix ≠ a supported **WebView2 host** path. Host apps call `CreateCoreWebView2Environment*` via loader → runtime; that chain is **more fragile** than launching Edge standalone. Do not assume Edge-in-Wine implies WebView2-in-Wine.

## Do we need to switch to building on Windows?

| Approach | Pros | Cons |
|----------|------|------|
| **Keep Linux + MinGW (current)** | Same as rest of repo; CI for compile-check; no MSYS2 on dev laptop required | WebView2 **runtime test** needs Windows VM/hardware; loader DLL bundling manual |
| **Add Windows MSYS2/VS job** | Native runtime test; can use `WebView2LoaderStatic.lib` with MSVC | Second build graph; splits maintainer docs |
| **Replace cross-build entirely** | Simplest WebView2 QA story | Abandons project’s Linux-first binding workflow |

**Verdict:** **Do not switch.** Extend Meson with optional `webview2-sdk` dependency and a Windows-only `meson` native build **option** for maintainers who want local runtime test without Wine — not a replacement for cross-build.

## Risks

| Risk | Mitigation |
|------|------------|
| SDK/runtime version skew | Pin SDK; document minimum Evergreen runtime; test `CompareBrowserVersions` in plumbing |
| COM callback lifetime bugs | Keep callbacks in C; minimal surface |
| Wine black screen | Document; do not block on Wine |
| Large runtime dependency | Document Evergreen install; no bundling in repo |
| Vala COM verbosity | C plumbing layer |
