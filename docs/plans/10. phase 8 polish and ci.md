### Scope (prioritized)

| Priority | Source | Valadoc entry points |
|----------|--------|----------------------|
| **P0** | `generated/win32-widgets.vala` | `Win32.Button`, `Win32.Edit`, `Win32.Window`, `Win32.WidgetDispatch`, … |
| **P1** | `generated/win32-ui-control-strings.vala` | `Win32.Ui.Controls.WC_*` (if included — large; may omit or index only) |

**🚫** Do not Valadoc the full ~42k-line monolith by default — too slow, low signal. Per-shard or widget-only packages match the app **`--pkg`** model.

## Packaging (binding consumption + app redistribution)

Phase 8 documents and automates **two packaging stories** — distinct from Valadoc/CI but equally user-facing.

### 1. Binding distribution (for Vala app authors)

vala.win32 ships **vapi shards + generated helpers + optional plumbing C** — not a linkable `libwin32`. App authors need a clear consumption path:

| Approach | Status | Notes |
|----------|--------|-------|
| **Git clone / submodule** + `--vapidir` | **✅** today | README quick-start; app `meson.build` points at repo `vapi/` + `generated/` |
| **Per-shard `--pkg`** (`win32-ui-windowsandmessaging`, …) | **✅** today | See [4. DONE - phase 2 ergonomic vapi.md](4.%20DONE%20-%20phase%202%20ergonomic%20vapi.md) naming rule |
| **`pkg-config` `.pc` per shard** | **⏳** | Deferred from Phase 2 (`win32-*.pc`); Phase 8 decides aggregate vs per-shard |
| **Meson subproject / wrap** | **⏳** | Optional `dependency('vala-win32')` for downstream apps |
| **Release tarball** (vapi + generated + metadata pin files) | **⏳** | No runtime binaries in binding tarball |

**Deliverables:**

- README **“Using vala.win32 in your app”** — minimum `--pkg` set for Track A (posix) vs Track B (gobject + `mingw-libs`), `--vapidir`, `--profile`, plumbing `.c` compile rules for WebView2
- Document **which shards** an ergonomic app needs vs raw vapi-only apps (coverage matrix cross-link)
- **⏳** Decide: ship `.pc` files, Meson subproject, or document clone-only for v1

### 2. Application redistribution (built `.exe` + DLLs)

Cross-compiled demos today copy some runtime files beside the exe; Phase 8 makes that **documented and scriptable**.

| Runtime piece | When required | Current copy path |
|---------------|---------------|-------------------|
| **MinGW GLib DLLs** | Track B ergonomic (`--profile=gobject`) | `meson.build` copies from `mingw-libs/` into `build/` |
| **`WebView2Loader.dll`** (x64) | WebView2 host / ergo demo | Meson copies from `build/vendor/webview2/x64/` |
| **Evergreen WebView2 Runtime** | WebView2 apps at run time | **Not** bundled in repo — OS install or [fixed-version folder](https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/distribution) |
| **comctl32 v6 manifest** | Themed common controls | Embedded in `ergonomic-widgets-demo`; other exes may need same pattern |
| **MSVC runtime** | If linking with MSVC later | Out of scope unless Windows-native job added |

**Deliverables:**

- **`scripts/package-app.sh`** (or `meson install` + custom target) — given a built exe name, gather exe + required DLLs + manifest into `dist/<app>/`
- README **redistribution checklist** — what to ship, what to document as a prerequisite install
- WebView2 section links [distribution docs](https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/distribution#files-to-ship-with-the-app) and states loader DLL bitness (x64 vs x86)
- **⏳** Optional: `make package-demo DEMO=hello-window` wrapper

### Verification

```bash
./scripts/package-app.sh build/hello-window.exe
# dist/hello-window/ contains exe + any copied DLLs; README lists missing OS prerequisites
```

- **⏳** README has binding-consumption and app-redistribution sections
- **⏳** At least one ergonomic demo and `webview2-host-demo` (or ergo twin) package cleanly via script
- **⏳** Documented minimum `--pkg` sets match what `compile-check` and examples actually use
