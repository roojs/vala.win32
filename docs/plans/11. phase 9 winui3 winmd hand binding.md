# Phase 9 WinUI3 WinMD and hand binding plan

## What the WebView2 generator can reuse

The current native generation path is useful up to the metadata workflow:

- vendored NuGet input under `build/vendor/`
- checked-in version pin under `metadata/*-ref.txt`
- symbol filtering with `metadata/filters/*.filter`
- `generate-binding` once metadata is available as win32json-shaped JSON

It does not directly apply past that point. WebView2 bindings come from a C COM
header (`WebView2.h`) that is scraped into JSON. WinUI3 is WinRT and starts from
binary `.winmd` metadata (`Microsoft.UI.Xaml.winmd`, `Microsoft.UI.Text.winmd`).
Those files describe runtime classes, activation factories, interfaces,
delegates, generics, properties, and events that the current `VapiEmitter` does
not model.

## Metadata scaffold added

- `scripts/vendor-winui3-sdk.sh` downloads `Microsoft.WindowsAppSDK.WinUI`
  and stages the WinMD/header files.
- `metadata/winui3-sdk-ref.txt` pins the NuGet version.
- `metadata/filters/winui3.filter` whitelists the first ABI symbols to bind.
- `metadata/winui3-api.files` is present but intentionally comments out JSON
  shards until a converter or hand-written seed shard exists.

## Recommended first hand binding

Start with a narrow ABI layer before attempting generated XAML coverage.

1. Add a hand vapi/header pair for the WinRT base ABI:
   - `IInspectable`
   - `IActivationFactory`
   - `HSTRING` helpers from `combase.dll`
   - `RoInitialize`, `RoUninitialize`, `RoGetActivationFactory`
2. Add the minimal WinUI3 ABI surface:
   - `Microsoft.UI.Xaml.Application`
   - `IApplicationFactory`
   - `IApplicationOverrides`
   - `LaunchActivatedEventArgs`
   - `Window` / `IWindow`
   - one or two controls (`Grid`, `TextBlock`) to prove content assignment
3. Add a native smoke example that:
   - initializes WinRT,
   - obtains the WinUI activation factory,
   - creates an `Application`,
   - creates and activates a `Window`,
   - tears down cleanly.
4. Only after the smoke example works, add a WinMD-to-JSON converter for the
   subset in `metadata/filters/winui3.filter`.

## Converter shape if generation follows

Emit win32json-compatible `Com` records for WinRT interfaces first, then extend
the parser/emitter for WinRT-specific concepts:

- `RuntimeClass` records with default interface and activatable factory metadata
- delegate records for WinRT events and async callbacks
- property accessors as COM methods
- generic type references such as `TypedEventHandler<TSender,TArgs>`
- WinRT string mapping (`HSTRING`) distinct from WebView2 `PWSTR`

Keep generated output separate from WebView2:

- `vapi/win32-ui-winui3.vapi`
- `src/win32-ui-winrt-base.*`
- `src/win32-ui-winui3-host.*`
- optional `generated/win32-ui-winui3-*` once the converter exists

## Validation gates

- `./scripts/vendor-winui3-sdk.sh` stages the pinned WinMDs on Linux and MSYS2.
- `meson setup build && meson compile -C build generate-binding` still succeeds.
- The first hand-bound example links against `ole32`, `runtimeobject`, and the
  Windows App SDK runtime/bootstrap libraries on Windows.
- Generated WinUI3 work should not be enabled in `regen` until the JSON shard is
  committed and the filter passes a compile-check.
